# PRD Issue Run

## Scope

This skill is scoped to the current Symphony Go repo root.

Use it when a Linear PRD parent issue already has child issues or explicit slices, and the user wants the whole PRD driven to closure by repeating:

```text
$triage 找到下一个可以处理的 issue
$tdd 推进
$tdd-acceptance-pr
```

This skill orchestrates those skills; their own detailed rules still apply. Prefer this skill over invoking `tdd-acceptance-pr` directly when the parent PRD and sub-issues already exist and the desired outcome is "keep going until everything is Done".

## Required Skill Files

Before executing the loop, read these files directly even if skill discovery is available:

- `$triage`: `.codex/skills/triage/SKILL.md`
- `$tdd`: `.codex/skills/tdd/SKILL.md`
- `$tdd-acceptance-pr`: `.codex/skills/tdd-acceptance-pr/SKILL.md`

Use the file contents as the authority for each phase.

## Core Rules

- Use `linear` CLI or `linear_graphql` for Linear writes. Do not use Linear MCP/app tools in unattended runs.
- Triage comments posted during triage must start with:
  `> *This was generated by AI during triage.*`
- Default to direct bounded execution from the current checkout unless the user asks to start the listener. Do not start a long-running listener just because this skill is invoked.
- Default to a commander/worker model: the main agent owns dependency graph assessment, subagent dispatch, PR merge ordering, root `main` sync, and parent closure; subagents own bounded child implementation work.
- Default to dependency-wave parallelism. Run the currently unblocked wave first; when a blocking child becomes Done, immediately re-read the child graph and dispatch all newly unblocked independent children with subagents. Merge and root `main` sync remain serialized.
- Use TDD vertically: one RED behavior test, minimal GREEN implementation, repeat.
- After TDD is green, use `tdd-acceptance-pr` for checklist evidence, Workpad update, PR merge, root `main` sync, and Linear Done verification.
- Maintain one `## Codex Workpad` per implementation issue.
- Do not infer new work from broad PRD user stories after all listed child issues are Done. If the parent child list or explicit checklist is complete, close the parent instead of inventing another slice.

## Preflight

From repo root:

```bash
pwd
git status --short --branch
git stash list
linear issue view <PRD_ISSUE>
linear issue comment list <PRD_ISSUE>
```

Read `AGENTS.md`, `docs/agents/issue-tracker.md`, and `docs/agents/triage-labels.md` before Linear writes.

### Main Preflight Commit

Before selecting or implementing any child issue, commit local discussion records, docs, skill edits, or other pre-existing local files to `main` so the PRD loop starts from an explicit baseline.

1. Inspect `git status --short --branch` and `git stash list`.
2. If the current branch is not `main`, preserve or finish its current work first; do not drag child-issue or feature changes onto `main` by accident.
3. On `main`, review dirty tracked and untracked files. If they are local notes, documentation, ADRs, skill updates, or other pre-loop records, add them and commit directly on `main` with a clear message such as `docs: record preflight PRD context`.
4. If the dirty files include source changes that look like unfinished child-issue implementation work, stop and ask how to classify them before committing.
5. If stashes exist, inspect `git stash show --stat` before the loop. Apply a stash to `main` only when it clearly contains pre-loop records that should be preserved. Commit the applied result on `main`, and drop the stash only after the commit is verified. If a stash conflicts or contains implementation work, leave it in place and record it in the parent PRD workpad.
6. After the preflight commit, verify `git status --short --branch` is clean on `main`, then fast-forward from `origin/main` or reconcile with remote before creating child branches/worktrees.
7. Record every preflight commit hash and any remaining stash in the parent PRD workpad.

Never silently delete, reset, or overwrite pre-existing dirty files or stashes. Do not start the child loop from a dirty `main`.

## Linear Read Protocol

Before selecting children or closing the parent, read [LINEAR-READING.md](LINEAR-READING.md). It records the exact CLI read pattern and the pitfalls from previous runs: table output truncates labels, `--all-teams` is often required, and the comments command is `linear issue comment list`.

## Parent PRD Assessment

Treat the PRD parent as a planning issue unless it has its own concrete implementation checklist.

Build the child set from `Sub-issues`, parent links, relations, comments, or Linear search for issues named by the PRD. Record identifier, title, state, labels, assignee, blockers, and Workpad status.

If any child is not Done, build a dependency and conflict graph before choosing work:

- node: child issue id, title, state, labels, assignee, branch, Workpad status, expected write scope;
- edge: `A -> B` when B is blocked by A;
- conflict edge: `A -- B` when A and B likely edit the same files, generated outputs, migrations, public interfaces, or shared tests;
- ready set: every non-Done child whose `Blocked by` issues are all terminal;
- runnable batch: the largest safe subset of the ready set with no conflict edges;
- downstream unlock set: children that should become ready after the current batch lands.

Classify each child before dispatch:

1. `ready-for-agent` / equivalent and no non-terminal `Blocked by` issues.
2. `needs-triage` / Backlog with enough detail to write an agent brief and no non-terminal `Blocked by` issues.
3. Unlabeled child with clear acceptance criteria and no non-terminal `Blocked by` issues.

Skip `ready-for-human`, `needs-info`, canceled, or Done children. Do not skip blocked children permanently; keep them in the dependency graph and re-evaluate after their blockers merge. If every remaining child is blocked by non-terminal issues, process the minimal blocker wave first. If every remaining child is skipped for human/info reasons, write the blocker summary to the parent PRD and stop.

## Commander And Dynamic DAG Scheduling

The main agent is the commander. It should spend time up front understanding the dependency DAG and likely file/module ownership, then dynamically allocate subagents. The commander's job is scheduling, integration, merge ordering, and parent closure; child implementation should be delegated whenever it is safe.

Scheduling loop:

1. Read all child issues, blockers, labels, descriptions, comments, and Workpads.
2. Infer a dependency DAG from Linear `Blocked by` relations first, then from explicit wording in issue text.
3. Infer expected write scope for each child from the package/module names, files mentioned, acceptance criteria, generated outputs, and prior child PRs.
4. Compute the current ready set: all non-Done children with no non-terminal blockers.
5. Partition the ready set into runnable batches by conflict risk. Prefer the largest batch of independent children, not simply the first issue.
6. Dispatch one subagent per child in the selected batch.
7. As subagents finish, review their result, merge PRs serially, sync `main`, and refresh branches/worktrees for still-running or queued children.
8. Re-read Linear and recompute the DAG after every merge, state change, or discovered blocker.
9. Repeat until every child is Done or the remaining graph is blocked by human/info constraints.

Examples:

- If six children exist and five are blocked by the first child, run the first child first. After it is Done, recompute the DAG and dispatch the five newly unblocked independent children together.
- If six children exist and the first five have no blockers while the sixth depends on all five, dispatch the first five together when their write scopes are independent. After they merge, dispatch the sixth.
- If several independent chains exist, dispatch the head of each chain together, then continue each chain as its predecessor lands.
- If one child changes a shared generated contract and several children consume that contract, treat the generated-contract child as a high-priority blocker even if Linear forgot the explicit relation; record the inferred dependency in the parent workpad.

Dispatch multiple child issues in the same runnable batch only if all of these are true:

- each selected child has no non-terminal `Blocked by` issue;
- their expected write scopes are disjoint;
- each child can own a separate branch or worktree;
- no child depends on another child's unmerged code;
- each worker is told that other agents may be editing the repo and that it must not revert others' changes.

Subagent prompt requirements:

- assign exactly one child issue and one branch/worktree;
- state the owned file/module scope and known sibling subagents;
- instruct the worker to edit files directly, use TDD, update the child Workpad, and report changed files, commands, PR URL, and blockers;
- instruct the worker not to merge to `main` unless explicitly assigned the serialized merge step by the commander.

Commander responsibilities:

1. Keep the parent dependency graph current after every merge or blocker change.
2. Avoid duplicate work between subagents.
3. Decide when to run a blocker personally versus delegating it; default to delegating implementation and keeping the main thread free for orchestration.
4. Review returned changes quickly before merge.
5. Serialize PR merge/root sync and update remaining branches from the new `origin/main`.
6. If two children touch overlapping files, stop parallelizing that pair and continue those children sequentially.
7. Record scheduling decisions, inferred dependencies, active subagents, queued children, and remaining blockers in the parent PRD workpad.

If subagents are unavailable in the current runtime, continue sequentially but record that runtime limitation in the parent PRD workpad.

## Child Issue Loop

For the selected child:

1. Triage: read issue/comments/labels/blockers/Workpad; post an agent brief if needed; assign self; keep exactly one category label and one state label; move into an active implementation state.
2. Create or reuse an issue branch/worktree from current `origin/main`. If running directly from the repo root, create a clean branch such as `symphony-go/<ISSUE>` and keep root `main` clean for sync/merge.
3. TDD: write one public-behavior RED test, implement minimal GREEN, repeat, then run relevant `./test.sh` and `./build.sh` wrappers.
4. Acceptance PR: re-read issue, tick only proven checklist items, update one Workpad, run `.codex/skills/pr/scripts/pr_merge_flow.sh` when in the expected issue worktree. If working directly from the root checkout and the PR script refuses that layout, perform the equivalent manual flow: commit, push branch, create PR with `Linear: <ISSUE>`, verify mergeability/checks, squash merge, fast-forward root `main`, and verify Linear Done.
5. Return to parent assessment and pick the next child.

## Parent Closure

When all listed child issues are Done, re-read the parent PRD and triage comments. Close the parent only when its explicit closure condition is proven, for example:

- parent remains the product/architecture source;
- each planned implementation slice is represented by a child issue;
- every listed child issue is Done;
- implementation kept the PRD's stated architecture boundary.

Post a parent closure comment with the triage disclaimer. Include the completed child list and why no further child issue is required. Then move the PRD parent to Done.

If the parent PRD mentions broad future ideas not represented by child issues, do not treat them as blockers. Say they need new explicit issues if the user wants to continue.

## Completion Report

Final response must include parent PRD id/state, child issues completed in this run, PR links and merge commits, validation commands that passed, and any remaining ideas intentionally not turned into work.
